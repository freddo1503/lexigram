= ADR-0001: Multi-Agent Architecture
:revdate: 2026-04-07
:status: accepted

== Context

Lexigram must transform raw French legislative text into three distinct outputs: a structured legal summary, an illustrative image, and an Instagram caption.
Each output requires a different skill set (legal analysis, visual design, social media marketing) and a specialized LLM — Mistral Large for all agents (text and images, see xref:0007-mistral-image-generation.adoc[ADR-0007]).

A single monolithic prompt cannot effectively handle all three concerns.
Separation allows each agent to have a specialized persona, backstory, and toolset.

== Decision

Use a *multi-agent architecture* with three specialized agents orchestrated in a *sequential pipeline*:

. *AnalysteJuridique* -- analyzes legal text, produces a structured summary
. *LexPictor* -- generates an illustrative image from the summary
. *LexMarker* -- writes an Instagram caption from the summary

Both LexPictor and LexMarker receive `text_summary` as context (fan-out from the first task).
Tasks execute sequentially in list order, but image_generation and caption do not depend on each other.

== Consequences

*Easier:*

* Each agent has a focused role, backstory, and prompt — easier to optimize independently.
* Different LLM models can be assigned per agent (currently all use Mistral Large — see xref:0007-mistral-image-generation.adoc[ADR-0007]).
* Adding a new output type (e.g., Twitter thread) means adding a new agent without modifying existing ones.

*Harder:*

* Sequential execution increases total latency (no parallelism).
* Debugging failures requires tracing through the agent chain.
* Agent-to-agent context passing must be carefully managed.

== Alternatives Considered

*Monolithic single-prompt approach:*
Rejected because a single prompt cannot effectively switch between legal analysis, image generation, and social media copywriting.
Different LLM models are optimal for each task.

*Parallel agent execution:*
Not feasible because the image and caption depend on the legal summary output.
The dependency chain is inherently sequential.

*Microservices architecture:*
Overkill for a batch job that runs once daily.
A single Lambda with in-process agents is simpler and cheaper.
